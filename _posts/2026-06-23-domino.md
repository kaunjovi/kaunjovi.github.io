## What are the critical stages of AI lifecycle?

1. **Design**, 
1. **Develop**, 
1. **Deploy**, and 
1. **Operate & Monitor**. 

executed in a  **iterative**—teams often loop back to earlier stages as new insights emerge.

Here is a breakdown of the key stages:

### Phase 1: Design (Problem Definition & Data Foundation)

*   **1. Problem Identification & Business Understanding**: This is the most critical stage. It involves clearly defining the business problem, determining if AI is the right solution, setting measurable objectives, and identifying key stakeholders.
*   **2. Data Gathering & Exploration**: You identify available data sources, assess data quality, and explore the data to gain initial insights. This stage confirms whether you have the necessary data to solve the defined problem.
*   **3. Data Wrangling & Preparation**: Raw data is cleaned, transformed, labeled, and formatted into a usable dataset for model training. This is often the most time-consuming part of the entire lifecycle.

### Phase 2: Develop (Model Building & Evaluation)

This is the core experimental phase where the model is created, tested, and refined.

*   **4. Model Engineering & Training**: Data scientists select algorithms and train multiple models, iteratively tuning hyperparameters to improve performance. This process can be computationally expensive and requires significant infrastructure.
*   **5. Model Evaluation & Validation**: The trained model is rigorously tested on new, unseen data to ensure it generalizes well and meets the predefined business and technical criteria. This stage also involves checking for fairness and bias.

### Phase 3: Deploy (Operationalization)

This phase moves the validated model from a development environment into production.

*   **6. Model Deployment & Integration**: The model is deployed into a production environment where it can process live data. This often involves integrating the model's predictions into existing business workflows or applications.

### Phase 4: Operate & Monitor (Ongoing Management)

The lifecycle does not end with deployment; it requires continuous oversight to maintain value.

*   **7. Monitoring & Maintenance**: Once live, the model's performance is continuously tracked. This is crucial for detecting **model drift**, where a model's accuracy degrades over time as the real-world data it processes changes. Continuous monitoring may lead to model retraining or refinement.
*   **8. (Optional) Retire**: Eventually, a model may become obsolete and need to be formally decommissioned and retired.

### Summary of Key Principles

*   **Iterative, Not Linear**: The lifecycle is a continuous loop. Insights from monitoring (Phase 4) can send you back to data preparation (Phase 1) or model training (Phase 2).
*   **Data is the Foundation**: The success of any AI project hinges on the quality and relevance of its data.
*   **Governance is Continuous**: Responsible AI requires ongoing attention to ethics, fairness, and security at every stage, not just at the beginning.


## What are the roles ? 

1. **Data Scientist** : Focuses on the research side. They explore the data, run statistical analyses, select algorithms, build initial models, and tune hyperparameters. Their primary output is a notebook or proof-of-concept model that works well on historical data.
1. **ML Engineer** : This is the bridge between Data Science and Software Engineering. They take the Data Scientist's experimental code and refactor it into production-grade, modular code. They write the training pipelines, implement feature engineering logic, and optimize the model code for speed and memory.

## The Open-Source Stack on AWS (Minimal Lock-in)

| Capability | Open-Source Component | AWS Role (Commodity Infrastructure) |
| :--- | :--- | :--- |
| **Orchestration & Pipelines** | ZenML (Flexible) or Kubeflow Pipelines (Comprehensive) | Runs on Amazon EKS (Kubernetes) or AWS Batch. ZenML pipelines can run locally and switch to AWS Batch/EKS with a simple config change. |
| **Experiment Tracking & Model Registry** | MLflow | Stores artifacts (models, logs) in Amazon S3. Stores metadata (params, metrics) in Amazon RDS (PostgreSQL) or Aurora. |
| **Model Serving (Real-time)** | KServe (Kubeflow) or Seldon Core | Deployed on Amazon EKS. Provides auto-scaling, canary releases, and A/B testing. Inference requests hit your open-source serving layer, not a proprietary AWS API. |
| **Feature Store** | Feast | Uses Amazon S3 as the offline feature store and Amazon DynamoDB (or Redis) as the online feature store. Feast abstracts the underlying storage so you can swap them later. |
| **Data Processing & Feature Engineering** | Snowflake (Snowpark) + Apache Spark (on EKS) | Snowflake handles heavy data transformations. If you want to avoid Snowflake lock-in, use Apache Spark on Amazon EMR on EKS—both are open-source, and EMR on EKS gives you managed Spark without proprietary APIs. |
| **CI/CD & Automation** | Apache Airflow or ZenML | Can run on Amazon EKS or AWS Fargate. While AWS offers CodePipeline, using Airflow/Terraform keeps your deployment logic portable. |
| **Infrastructure as Code (IaC)** | Terraform | Provision all AWS resources (EKS clusters, S3 buckets, RDS databases) declaratively. This is critical for avoiding cloud management lock-in. |


| Component | Decision |
| :--- | :--- |
| **Compute Layer** | Amazon EKS (Kubernetes) – you are not locked in because Kubernetes runs everywhere. |
| **Orchestration** | ZenML (most portable) or Kubeflow (Kubernetes-native). |
| **Model Serving** | KServe (open-source, runs on any K8s). |
| **Tracking & Registry** | MLflow (open-source, storage backend is just S3 + RDS). |
| **Feature Store** | Feast (open-source, storage backends are S3 and DynamoDB/Redis). |
| **Data Warehouse** | Snowflake (proprietary but replaceable) or Apache Iceberg on S3 (fully open-source). |
| **Infrastructure Management** | Terraform – essential for cloud-agnostic provisioning. |
